1. Visao Geral da Arquitetura

Stack: Node.js + Express + Prisma (PostgreSQL) | React 18 + Vite + React Router v6

Autenticacao: JWT com usuario/senha unico via .env (ADMIN_USER / ADMIN_PASS)

Estrutura: MVC classico — routes → controllers → services → Prisma ORM

Entidades: 12 models Prisma, 9 controllers, 11 arquivos de rota, 2 services de dominio

Fluxo de negocio: Cliente → Base Alimentar → Pedido → Gerar Cardapio → Aprovacao → Ordem de Producao

Pontos Positivos

  • Separacao clara entre camadas (routes / controllers / services)
  • Uso correto de transactions Prisma em operacoes criticas (criar/atualizar pedido, persistir versao)
  • Estado de pedido bem modelado com enum e maquina de transicoes explicita (TRANSICOES)
  • Error handler centralizado com tratamento especifico de erros Prisma
  • Singleton do PrismaClient com cache global em dev (evita connection leak com hot reload)

2. Code Review — Checklist 7 Dimensoes

2.1 Seguranca

BLOQUEANTE Sem rate limiting em nenhum endpoint
Seguranca auth.routes.js portal.routes.js

O endpoint POST /api/auth/login e a rota publica POST /api/portal/solicitar nao possuem rate limiting. Um atacante pode executar brute force no login ou flood de solicitacoes no portal publico sem nenhum limite.

Impacto: Comprometimento de credenciais via brute force; DDoS no portal publico; spam de solicitacoes falsas.

Correcao:

// npm install express-rate-limit
const rateLimit = require('express-rate-limit');

// Login: max 5 tentativas a cada 15 min
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  message: { error: 'Muitas tentativas. Tente novamente em 15 minutos.' }
});
app.use('/api/auth/login', loginLimiter);

// Portal publico: max 10 solicitacoes/hora por IP
const portalLimiter = rateLimit({
  windowMs: 60 * 60 * 1000,
  max: 10,
  message: { error: 'Limite de solicitacoes atingido.' }
});
app.use('/api/portal/solicitar', portalLimiter);
BLOQUEANTE Ausencia total de headers de seguranca (Helmet)
Seguranca app.js

O servidor Express nao usa helmet. Isso expoe o app a ataques XSS, clickjacking, MIME sniffing, e vaza a assinatura X-Powered-By: Express em todo response.

Correcao:

// npm install helmet
const helmet = require('helmet');
app.use(helmet());
BLOQUEANTE CORS totalmente aberto
Seguranca app.js

app.use(cors()) sem nenhuma config aceita requests de QUALQUER origem. Em producao, qualquer site malicioso pode fazer requests autenticados se o usuario tiver um token valido.

app.use(cors({
  origin: process.env.FRONTEND_URL || 'http://localhost:5173',
  credentials: true
}));
BLOQUEANTE Autenticacao com credencial unica hardcoded em .env
Seguranca auth.routes.js

O sistema usa um unico usuario/senha definido em variaveis de ambiente (ADMIN_USER / ADMIN_PASS). Nao ha hashing de senha — a comparacao e direta em texto plano. Se alguem tiver acesso ao .env (deploy leak, log acidental), tem acesso total.

Para o momento (MVP): Aceitavel se: (a) o .env nao esta commitado, (b) a senha e forte, (c) ha rate limiting no login. Mas para producao seria necessario um model User com bcrypt.

ALTA Vazamento de detalhes de erro em producao
Seguranca errorHandler.js

O PrismaClientValidationError retorna err.message completo ao cliente, que pode conter nomes de campos, tipos e detalhes internos do schema. Em producao, isso vaza informacoes do banco.

if (err.name === 'PrismaClientValidationError') {
  return res.status(400).json({
    error: 'Dados invalidos fornecidos.',
    // NAO retornar err.message em producao
    ...(process.env.NODE_ENV !== 'production' && { details: err.message }),
  });
}

2.2 Performance

ALTA Listagens sem paginacao
Performance Todos controllers

Todos os endpoints GET / (listar) fazem findMany() sem take/skip. Com 500+ clientes, 1000+ pedidos, ou 5000+ preparos, a resposta vai crescer indefinidamente, causando lentidao e potencialmente timeout.

Prioridade: pedidos.listar e o mais critico — inclui joins com cliente, aprovacao, ordemProducao e count de versoes.

const listar = async (req, res, next) => {
  try {
    const page = Math.max(1, parseInt(req.query.page) || 1);
    const limit = Math.min(100, Math.max(1, parseInt(req.query.limit) || 20));
    const skip = (page - 1) * limit;

    const [data, total] = await Promise.all([
      prisma.pedidoDieta.findMany({
        skip, take: limit,
        orderBy: { createdAt: 'desc' },
        include: { /* ... */ },
      }),
      prisma.pedidoDieta.count(),
    ]);
    res.json({ data, total, page, limit, totalPages: Math.ceil(total / limit) });
  } catch (err) { next(err); }
};
MEDIA Deep include sem selecao de campos em pedidos.buscarPorId
Performance pedidos.controller.js

A funcao includeDetalhe() faz 6 niveis de include aninhado (pedido → versoes → lotes → itens → preparo → alimento → grupo). Para pedidos com muitas versoes e lotes, isso pode gerar queries N+1 implicitas e payloads enormes.

Sugestao: Adicionar select nos niveis mais profundos e considerar carregar versoes sob demanda (lazy load via endpoint separado).

MEDIA Loop sequencial em gerarOrdem (N queries para itens)
Performance ordemProducao.service.js

A gerarOrdem ja usa um unico query com includes, o que e bom. Porem, o create da ordem e o update do pedido sao duas operacoes separadas sem transaction — se o update falhar, a ordem fica criada mas o pedido nao muda de status.

2.3 Funcionalidade e Edge Cases

ALTA Race condition: gerarOrdem sem transaction
Bug Potencial ordemProducao.service.js

A funcao gerarOrdem() faz prisma.ordemProducao.create() e depois prisma.pedidoDieta.update() em dois comandos separados. Se o segundo falha, a ordem fica criada mas o pedido nao vai para EM_PRODUCAO.

Alem disso, o controller ordens.gerar() verifica se ja existe ordem ANTES de chamar o service, mas dois requests simultaneos podem passar na checagem ao mesmo tempo (race condition).

// Correcao: wrapping em transaction
async function gerarOrdem(pedidoId) {
  return prisma.$transaction(async (tx) => {
    // Lock: tenta criar a ordem (unique constraint protege contra duplicata)
    const ordem = await tx.ordemProducao.create({
      data: { pedidoId, itensConsolidados },
    });
    await tx.pedidoDieta.update({
      where: { id: pedidoId },
      data: { status: 'EM_PRODUCAO' },
    });
    return ordem;
  });
}
ALTA Race condition: atualizarStatus sem lock otimista
Bug Potencial ordens.controller.js

Em ordens.atualizarStatus(), o update do status da ordem e o update do pedido sao operacoes separadas. Se dois usuarios clicarem "Concluir" ao mesmo tempo, ambos podem passar na verificacao.

Sugestao: Usar $transaction e idealmente lock otimista com updatedAt.

ALTA Delete de cliente nao verifica dependencias
Bug Potencial clientes.controller.js

O endpoint DELETE /api/clientes/:id faz prisma.cliente.delete() diretamente. Se o cliente tem pedidos, o Prisma vai lancar um erro de foreign key (P2003) que nao e tratado no errorHandler — o usuario recebe erro generico 500.

Correcao: Verificar pedidos antes de deletar ou adicionar tratamento de P2003.

// No errorHandler.js, adicionar:
if (err.code === 'P2003') {
  return res.status(409).json({
    error: 'Nao e possivel excluir: existem registros dependentes.',
    details: err.meta,
  });
}
MEDIA Cardapio pode entrar em loop infinito com combinacoes insuficientes
Bug Potencial cardapio.service.js:85

O loop for (let t = 0; combos.length < N && t < N * 2000; t++) tem um limite de iteracoes, o que e bom. Porem, se nao conseguir gerar combos unicos suficientes, o fallback na linha 110 (while (combos.length < N) combos.push(combos[(combos.length - 1) % ...])) pode ter combos.length === 0, causando divisao por zero em Math.max(combos.length, 1) e duplicando undefined.

Na pratica: Se preparos.length === 0, ja retorna [] antes. Mas se nCarb === 0 && nLeg === 0 && nVeg === 0 e nProt > 0, combos vao ter chaves duplicadas e podem nao gerar combos unicos suficientes.

MEDIA Inconsistencia de grupoNome: "Legumes" vs "Legume"
Bug Potencial cardapio.service.js / pedidos.controller.js

No controller de pedidos, o grupo e definido como 'Legumes' (plural). No service de cardapio, a funcao montarOpcoesSimples tambem filtra por 'Legumes'. Mas no schema Prisma e docs, aparece como "Legume" em singular (legume no SolicitacaoOrcamento). Qualquer inconsistencia vai fazer o filtro nao encontrar itens.

2.4 Legibilidade

BOM Codigo bem organizado e consistente

A estrutura MVC esta limpa, os nomes de funcoes sao descritivos em portugues (coerente com o dominio), e o service de cardapio tem comentarios claros explicando o algoritmo TACO. O padrao try/catch/next e consistente em todos controllers.

BAIXA Constantes magicas no algoritmo de cardapio
Legibilidade cardapio.service.js:79

PRIMO_A = 37, PRIMO_B = 53 e N * 2000 sao constantes magicas sem explicacao do por que esses valores foram escolhidos. Um comentario breve ajudaria a manutencao futura.

2.6 Arquitetura

MEDIA Logica de negocio espalhada nos controllers
Arquitetura

As regras de transicao de status, validacoes de negocio e operacoes compostas estao nos controllers (pedidos.controller.js, ordens.controller.js, aprovacoes.controller.js). Idealmente, cada dominio teria um service dedicado (PedidoService, AprovacaoService) com toda logica de negocio, e o controller apenas faria parse do request e chamaria o service.

MEDIA Sem validacao de input estruturada
Arquitetura

A validacao de input e feita manualmente em cada controller com if (!campo). Isso e propenso a esquecimentos e nao valida tipos/formatos. Uma lib como zod ou joi centralizaria isso.

// Exemplo com Zod
const { z } = require('zod');

const criarPedidoSchema = z.object({
  clienteId:     z.string().cuid(),
  totalPratos:   z.number().int().min(5),
  maxRepeticoes: z.number().int().min(1),
  proteinas:     z.array(z.object({
    alimentoBaseId:   z.string().cuid(),
    gramagem:         z.number().positive(),
    quantidadePratos: z.number().int().positive(),
  })).min(1),
});

2.7 Error Handling

MEDIA console.error sem estrutura (nao e pesquisavel em logs)
Error Handling errorHandler.js

O error handler faz console.error('Error:', err) que loga o objeto inteiro. Em producao, isso nao e pesquisavel, nao tem correlation ID, e mistura erros 400 (user error) com erros 500 (server error). Considere usar um logger estruturado como pino ou winston.

3. Bugs Potenciais — Debugging Sistematico

Aplicando o framework de hipoteses para bugs latentes no sistema:

Bug #1: Ordem de Producao duplicada (Race Condition)

CampoDetalhe
HipoteseDois cliques rapidos no botao "Gerar Ordem" podem criar duas ordens para o mesmo pedido
CamadaController → Service (backend)
Causa raizChecagem de existencia e criacao nao sao atomicas. O controller verifica findUnique e depois chama o service. Dois requests podem passar pela checagem simultaneamente.
Probabilidade80% em uso real (botao sem debounce + rede lenta)
Como testarEnviar 2 requests POST simultaneos para /api/ordens-producao/:pedidoId
Protecao atualConstraint @unique no pedidoId vai lancar P2002 no segundo request — cai no error handler como "registro duplicado" (409). O frontend pode nao tratar esse caso graciosamente.
CorrecaoUsar transaction + tratar P2002 no controller como resposta amigavel

Bug #2: Cardapio gerado com pratos identicos

CampoDetalhe
HipoteseQuando ha poucas opcoes de carboidrato/leguminosa/legume, o algoritmo gera lotes com combinacoes identicas
CamadaService (cardapio.service.js)
Causa raizO fallback na linha 110 duplica combos quando nao consegue gerar combinacoes unicas suficientes. Se nProt=1, nCarb=1, nLeg=1, nVeg=1, so existe 1 combo possivel — todos os lotes terao o mesmo prato.
Probabilidade60% para pedidos com poucas opcoes selecionadas
Como testarCriar pedido com 10 pratos, 1 proteina, 1 carboidrato, 1 leguminosa, 1 legume, maxRepeticoes=2
ImpactoCliente recebe cardapio monotono, recusa a aprovacao
CorrecaoAdicionar warning quando diversidade e insuficiente para os parametros do pedido

Bug #3: Delete em cascata pode destruir dados em producao

CampoDetalhe
HipoteseDeletar um AlimentoBase apaga todos seus preparos (onDelete: Cascade), que podem estar referenciados em ItemLote de pedidos ativos
CamadaSchema Prisma + Controller
Causa raizAlimentoBase → PreparoAlimento tem onDelete: Cascade, mas ItemLote referencia PreparoAlimento sem cascade. Deletar o alimento vai falhar com P2003 se houver itens de lote referenciando o preparo — mas a mensagem de erro nao e clara.
Probabilidade40% em uso normal
CorrecaoPreferir soft-delete (campo ativo: false) ao inves de hard delete para alimentos e preparos que ja estao em uso

Bug #4: Pedido editado apos cardapio gerado perde a versao

CampoDetalhe
HipoteseAo editar um pedido com status GERADO, o controller reseta para PENDENTE e deleta proteinas/itens permitidos, mas NAO deleta as versoes de cardapio ja geradas. As versoes antigas ficam orfas com dados incompativeis.
CamadaController (pedidos.controller.js:216-258)
Causa raizO atualizar() deleta e recria proteinas e itensPermitidos, mas nao toca em CardapioVersao. As versoes antigas referenciam preparos que podem nao fazer mais parte do pedido.
Probabilidade70% — workflow muito comum (gerar, ajustar, re-gerar)
CorrecaoNo atualizar(), desativar versoes anteriores ou deletar completamente quando o pedido e re-configurado

4.2 Corrigir race condition em gerarOrdem

// ordemProducao.service.js - wrapping completo em transaction
async function gerarOrdem(pedidoId) {
  return prisma.$transaction(async (tx) => {
    const versaoAtiva = await tx.cardapioVersao.findFirst({
      where: { pedidoId, ativo: true },
      include: { lotes: { include: { itens: { include: { preparo: {
        include: { alimento: { include: { grupo: true } } }
      }}}}}},
      orderBy: { numero: 'desc' },
    });

    if (!versaoAtiva || versaoAtiva.lotes.length === 0)
      throw new Error('Nenhuma versao de cardapio ativa encontrada.');

    // ... (logica de consolidacao existente) ...

    const ordem = await tx.ordemProducao.create({
      data: { pedidoId, itensConsolidados },
    });
    await tx.pedidoDieta.update({
      where: { id: pedidoId },
      data: { status: 'EM_PRODUCAO' },
    });
    return ordem;
  });
}
